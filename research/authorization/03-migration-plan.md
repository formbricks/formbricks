# Migration Plan â€” getting today's behavior onto SpiceDB (Scope 1)

_Objective / audience._ This is the technical plan for **Scope 1** of the authorization redesign: getting Formbricks' current access approach running on **SpiceDB** with no observable behavior change, then being positioned to progressively refactor it. Audience: engineering. It assumes the entity map and enforcement inventory from the Status Quo doc and the target schema from the Target Model doc (the runnable companion `./spicedb-schema-draft.zed`).

**Scope framing.** Scope 1 is complete when the current access approach is running on SpiceDB and we can progressively refactor it to meet BI/product requirements. **Phase 1 is PLUMBING ONLY** â€” no product-model changes: contacts stay workspace-scoped, no per-object sharing yet, no deployment object, no owner feature. Those are product direction (see Product Lens) and are explicitly deferred; they belong to the progressive-refactor phases below, not to Scope 1.

**Phase 0 â€” one choke point (no behavior change).** Build `can(actor, action, resource)` in `apps/web/lib/authz/` wrapping _today's_ logic; migrate all enforcement sites (see Status Quo, "Where authorization is enforced today") onto it; delete scattered helpers. This is prerequisite work regardless of engine and immediately fixes I-13/I-14 drift risk. Emit decision logs from day one.

## Migration phasing (Scope 1 = Phases 0â€“2)

**Phase 0 â€” one choke point (no behavior change).** As above: `can(actor, action, resource)` in `apps/web/lib/authz/` wrapping today's logic, every enforcement site (see Status Quo, "Where authorization is enforced today") routed through it, scattered helpers deleted, decision logs from day one. This is engine-independent groundwork and the place where the [prerequisite] refactors below land.

**Phase 1 â€” backfill today's grants into SpiceDB and run in shadow mode (no behavior change).** Ship SpiceDB (compose/helm/managed) with the schema from the Target Model draft â€” the runnable `./spicedb-schema-draft.zed`. Backfill every existing grant via `ImportBulk` from the **five Prisma grant tables** (`Membership`, `TeamUser`, `WorkspaceTeam`, `ApiKeyWorkspace`, `FeedbackDirectoryWorkspace`). The mapping is deterministic and mechanical â€” the Day-0 parity table at the bottom of the `.zed` file: `Membership` â†’ org relations, `TeamUser` â†’ team relations, `WorkspaceTeam.read/readWrite/manage` â†’ workspace `viewer/editor/manager` with `team#member` subjects, `ApiKeyWorkspace` â†’ the same with `service_account` subjects, `FeedbackDirectoryWorkspace` â†’ `feedback_dataset#workspace_access`. Dual-write on every grant mutation (Prisma tx + outbox â†’ `WriteRelationships`). `can()` asks both engines, **enforces legacy**, and logs diffs until they reach zero. This shadow comparison is the parity gate for the Phase 1â†’2 transition â€” see the Shadow-Mode Proposal (`./04-shadow-mode-proposal.md`) for the harness and diff methodology.

**Phase 2 â€” flip enforcement, surface by surface (cutover).** Once shadow diffs are zero, flip `can()` to enforce SpiceDB. Feature-flag per route group in order: server actions â†’ v2 API â†’ pages â†’ v1. Keep the Prisma grant tables as the source of truth for rebuild/reconciliation (nightly `ExportBulk` diff job). At the end of Phase 2, **Scope 1 is complete**: today's exact access approach runs on SpiceDB, and the codebase can be progressively refactored toward BI/product requirements without a second migration of the plumbing.

### Model-consistency problems (Â§3.B)

- **[prerequisite]** **I-10 Five permission vocabularies** that don't compose: org roles, team roles (`admin|contributor`), workspace-team permissions (`read|readWrite|manage`), API-key permissions (`read|write|manage` + `organizationAccess`), OAuth scopes (`surveys:read|write`). v3 already hard-codes a translation table between the key ladder and the team ladder (`writeâ†’readWrite`, [v3/lib/auth.ts:14](../../apps/web/app/api/v3/lib/auth.ts)) â€” drift between ladders is a latent bug class.
- **[prerequisite]** **I-11 Org owner/manager implicit bypass, re-implemented everywhere.** At least 5 distinct server-side code paths + 29 inline `isOwner || isManager` component checks (sprawl metrics in Status Quo, "Where authorization is enforced today"). Divergence between them is invisible until it's an incident.
- **[prerequisite]** **I-12 Teams are overloaded.** Simultaneously people-groups and the _only_ grant mechanism â€” real-world team structure and access structure can't diverge without one-person-team hacks.
- **[prerequisite]** **I-13 Enforcement is scattered across seven stacks** (see Status Quo, "Where authorization is enforced today") with no central `can()`. v3's `TV3Authentication` union is the first normalization attempt â€” Phase 0 should generalize it.
- **[prerequisite]** **I-14 UI-gating â‰  enforcement, with footguns.** `getTeamPermissionFlags` returns **mutually exclusive** booleans â€” a `manage` user has `hasReadAccess === false` ([teams/utils/teams.ts:25](../../apps/web/modules/ee/teams/utils/teams.ts)); `getWorkspaceAuth`'s `isReadOnly = isMember && hasReadAccess` depends on this quirk. Components gating on `hasReadAccess` alone silently mis-handle higher roles.
- **[prerequisite]** **I-15 License-gating couples entitlement with authorization.** `accessControl` (EE) gates role management; without a license all EE features are `false` (roles in the DB persist â€” losing a license does _not_ make everyone owner, verified). CE and EE differ in code paths, not data. The target model should make entitlements an app-level concern cleanly separated from grants (Q10).
- **[prerequisite]** **I-16 Weakly-modeled references, weakly enforced.** `Webhook.surveyIds: String[]` (no FK): v1 stores them **verbatim with no workspace check** ([v1/webhooks/lib/webhook.ts:23](../../apps/web/app/api/v1/webhooks/lib/webhook.ts)); v2 checks all ids share _one_ workspace but **never compares it to the webhook's authorized workspace** ([v2/management/webhooks/route.ts:65](../../apps/web/modules/api/v2/management/webhooks/route.ts)) â€” a key holder can attach another workspace's surveys to a webhook. `Invite.teamIds: String[]` follows the same pattern.
- **[prerequisite]** **I-17 Dead vocabulary lying around.** Legacy `ZMembershipRole` (`owner/admin/editor/developer/viewer`) still exported next to the live `ZOrganizationRole` ([memberships.ts:3](../../packages/types/memberships.ts)); `hasOrganizationAuthority`/`hasOrganizationOwnership`/`isOwner`/`isManagerOrOwner` appear test-only; `hasPermission`/`hasWorkspacePermission` are byte-identical duplicates. Cleanup belongs in Phase 0.

### Public-surface & machine-credential weaknesses (Â§3.C)

- **[standalone â†’ see Non-SpiceDB Issues doc]** **I-18 Unauthenticated response mutation.** `PUT /api/v1|v2/client/[workspaceId]/responses/[responseId]` has **no per-response ownership or token check** â€” anyone knowing a responseId + workspaceId can append data to any _unfinished_ response ([put-response-handler.ts:226](<../../apps/web/app/api/v1/client/[workspaceId]/responses/[responseId]/lib/put-response-handler.ts>)). Needs a capability token (e.g. suid/display-bound) regardless of the SpiceDB work.
- **[prerequisite]** **I-19 Workspace id = catalog read.** `GET /api/v1/client/[workspaceId]/environment` returns _all_ in-progress app-survey definitions (questions, triggers, styling) to anyone holding a workspace id **or legacy environment id** (widely distributed in old embeds). Segment filters are sanitized (good â€” only `hasFilters` leaks). By design for the SDK, but should be modeled as an explicit _public/anonymous_ relation in the target model, not an implicit bypass.
- **[standalone â†’ see Non-SpiceDB Issues doc]** **I-20 Personal links are effectively non-revocable** â€” no per-link store; only optional JWT expiry or rotating the global `ENCRYPTION_KEY` (which invalidates _every_ link, single-use token and more). Contact deletion doesn't invalidate issued JWTs (render-time lookups just fail).
- **[standalone â†’ see Non-SpiceDB Issues doc]** **I-21 MCP/OAuth blast radius = the whole user.** Delegated tokens carry only `surveys:read|write` scopes; resource authorization is _entirely_ the user's memberships â€” no per-resource consent or narrowing ([modules/mcp/auth.ts:298](../../apps/web/modules/mcp/auth.ts)). Dynamic Client Registration is open + unauthenticated (rate-limited only, [mcp-oauth-provider-options.ts:21](../../apps/web/modules/auth/lib/mcp-oauth-provider-options.ts)). Per-object grants would let consent screens offer "this survey only".
- **[prerequisite]** **I-22 Hub trust model is app-side only.** One global `HUB_API_KEY`; `tenant_id` is caller-supplied per request ([hub-client.ts:20](../../apps/web/modules/hub/hub-client.ts)). The Prisma assignment tables + app checks are the _only_ thing preventing cross-tenant reads. Same story for Cube (`CUBEJS_API_SECRET`-signed JWT, good pattern â€” but directory-scoped only, see I-3).
- **[standalone â†’ see Non-SpiceDB Issues doc]** **I-23** `CRON_SECRET` **is defined but no enforcement site was found** in this tree (grep) â€” verify cron-route protection exists somewhere before assuming it.
- **[standalone â†’ see Non-SpiceDB Issues doc]** **I-24 Secrets-at-rest inconsistency** (adjacent, worth a ticket): integration OAuth tokens for Google/Airtable/Slack and webhook secrets appear stored **unencrypted** in `config`/`secret`; only Notion, 2FA and link tokens use `ENCRYPTION_KEY`/`BETTER_AUTH_SECRET`. Whoever holds workspace access (or DB access) holds live third-party tokens. **Verified 2026-07-12:** [googleSheet/service.ts:186-199](../../apps/web/lib/googleSheet/service.ts) writes `refresh_token`/`access_token` into `config.key` verbatim, whereas [notion/callback/route.ts:117](<../../apps/web/app/api/v1/integrations/notion/callback/route.ts>) does `symmetricEncrypt(tokenData.access_token, ENCRYPTION_KEY)` â€” a DB dump leaks usable long-lived Google refresh tokens. High-severity; hotfix candidate.
