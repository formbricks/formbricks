# Offline Split Response Runbook

Use this when a kiosk or client reports that a completed offline-supported survey is "missing" answers after restore.

## What to look for

- Same survey and same `offlineSupport=true` URL.
- Same user agent or device profile.
- Two responses created within a short window.
- One unfinished partial response followed by one finished partial response.
- At least one overlapping identical answer or identical TTC value across both responses.

## How to validate

1. Pull both response rows from the database for the affected survey and time window.
2. Compare `data`, `ttc`, `displayId`, `createdAt`, `updatedAt`, and `finished`.
3. Confirm that early blocks live on one response and later blocks live on the other.
4. Check backend logs for the old `"Response is already finished"` / `400 PUT` signature. If that signature is absent, treat this as the restore-anchor split-response issue instead.

## Default remediation

- Preserve both source rows.
- Merge the answer sets manually for customer-facing export or analysis.
- Note the paired response IDs in the incident record so support and engineering can cross-reference the split submission later.
