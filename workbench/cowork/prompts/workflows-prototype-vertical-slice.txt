Non-negotiable PoC architecture:
- Workflows are API-first. Use minimal v3 route handlers; do not use Next.js server actions as the
  workflow backend contract.
- The dashboard is a client of the v3 API. No dashboard-only workflow behavior.
- Workflow definitions are JSON documents validated by shared client-safe zod schemas.
- The backend validates the same payload again and remains authoritative.
- Keep workflow definition records and workflow run records separate.
- Use BullMQ under `packages/jobs` for execution. Do not execute workflow side effects
  synchronously in the response request path.
- React Flow state is derived from workflow JSON; it is not a separate source of truth.
- Jotai is acceptable only for workflow-builder editor state such as selected node, open drawer,
  dirty draft, local validation issues, and canvas UI state. Keep API/server state in the v3 API/query
  layer.
- Use workflow definition lifecycle values: `draft`, `active`, `disabled`.
- Use workflow run statuses: `queued`, `running`, `completed`, `failed`, `canceled`.

PoC slice:
- One trigger: Response Completed.
- One flow node: If/Else.
- Two actions: Send Email preview and Send Webhook preview with the fixed envelope
  `{ response: <previous output or trigger payload> }`.
- Minimal v3 API: workflow CRUD, activate/disable, and run list/detail. Trigger enqueue is a server-only
  service called from the response pipeline, not an externally reachable HTTP route.
- One local BullMQ queue and worker.
- Dashboard: workflows list, React Flow canvas, fixed/collapsible drawer, run list, run detail.
- No real email or webhook calls.

Readiness decisions from 001-000:
- Use `responseFinished` in the existing response pipeline as the Response Completed hook.
- Add Workflows as its own main dashboard sidebar section for the PoC.
- Put shared workflow schemas in a browser-safe `@formbricks/types` workflow module.
- Add separate `Workflow` and `WorkflowRun` persistence.
- Relate `WorkflowRun` to `Workflow` through `[workflowId, workspaceId]`, not `workflowId` alone.
- Add a `workflow-run.process` BullMQ job with PoC attempts set to `1`, not the package default retries.

Definition of PoC done:
- A developer can create a workflow through the v3 API and see/edit it in the dashboard.
- The dashboard validates workflow JSON locally with shared zod schemas, and the API validates again.
- A developer can activate and disable the workflow through v3-backed controls.
- A Response Completed event creates a run for matching active workflows.
- A BullMQ worker evaluates If/Else and produces no-send Email/Webhook preview outputs.
- Run list/detail show status, timestamps, step outputs, and a failure message.
- Disabling the workflow prevents new runs.
- Tests cover the schema, graph execution, minimal API behavior, and worker status transition path.
- No real email or webhook call is sent.
