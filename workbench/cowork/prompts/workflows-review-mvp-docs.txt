        ## Basics:

        Inspired by Twenty workflows:

        - examples of inputs/entry points/triggers: Response Received, Survey Completed, Webhook Received (not for MVP). etc.
        - action types for mvp: Compute/Calculate (using data from the trigger or the result of a previous action) probably we will need a math interpreter or some tool for this to be good, If/Else branching

        - Actions not needed for mvp but to be considered for a v2 : loop/iterator of collections, delays/postpone  the rest of the actions, Human step/input (form)

        - Good to have for MVP (depending on time): AI Agent action (prompt with vars -> output, model selector)

        ## MVP low-hanging-fruit

        We have already external service connectors such as Notion, Webhooks, etc.

        We also have the Follow-ups functionality which sends emails automatically "Send automatic follow-ups. Send messages to respondents, yourself or team mates.": it's actually
        a workflow, but without any conditions or logic in the middle: it is always triggered.

        These two parts are very good candidates to bring into the MVP: Webhooks and Follow-ups (that we can rename to simply Send Email).
        With these 2 we can be very satisfied.
        Also, a good nice to have will be an action that asks AI in the middle and transforms returns an answer that can be passed to the next action.


        ## Tech behind:

        - xyflow / React Flow for the diagrams (a canvas layout)
        - We want to use BullMQ for queues and create the job system in packages/jobs
        - It has to be integrated in the dashboard Next.js app, under so tailwind, shadcn, radix, etc.
        - In the menu it will be under Unify Feeback, under Dashboards.
        - We don't need to show versioning in the UI in the MVP but it's good that internally we are prepared
        - A workflow is a JSON document. We should define data model with a similar approach as we did in this prototype https://github.com/itsjavi/formbricks-workflows/tree/main/app/lib/workflows
        - Since a workflow is a JSON document, it can be generated by LLMs given a schema and rules (e.g zod)
        - Workflow data and workflow config live in separate tables as JSONB documents in PGSQL. They belong to the workspace and inherit the same permissions (e.g. data that a workspace can see, the workflow will be able to use)
        - Every completed action, updates the relevant data for that workflow run -> every action computed/final output is saved in the doc
        - We need logs for the workflow runs, to know the status why they failed, how much time they took, etc.
        - Audit logs is important as well (start simple)
        - We need a UI to see the configured workfows and the workflow runs (so proably both make sense to be in the main sidebar menu, under its own group "Workflows" like unify feedback has).
        - In the workflow runs page we should be able to see a table with them and their status, paginated. On click, we sohuld be able to see the metadata in detail, dates, logs?

        IMPORTANT BE decisions and constraints:
        - we don't want to use Next.js actions for this.
        - everything that can be done via the UI should also be able to be done via the v3 API (the openapi spec is in the /docs folder)
        - before coding the UI we will need a list of API endpoints that we will need, and plan the missing ones properly (part of milestone will be that, to finish the API requirements)
        - the UI will just become a client of the API.
        - The main reason whe we want to do this is because we want to open the door to LLMs and MCPs to use our API directly to create and manage the workflows, plus that we are moving away from nextjs server actions.



        ## To consider / risks

        Actions should probably support structured outputs. But, since we know the structure of our triggers input/outputs (survey answers etc.), for the MVP we dont need to rollout a data mapper, or schema creator tool. For the computed outputs we can infer their type depending on the operation (we will only have math ops or string ops).

        Things to consider/risks (that might be part of MVP or not depending on complexity):
        - Users having customers filling surveys like crazy and for every answer we trigger a workflow run. Multiply that for thousands of orgs and workspaces
        that we have in the cloud version. Somehow it would be nice to be able to buffer them within a time-window and process them in batches.
        - Webhook/API calls will be unauthenticated by default. We won't support encrypting headers/secrets in the MVP and that should be clear in the UI.
        - We probably need to track webhook call failures / latency / elapsed time, etc. They shouldn't bring our job workers down or be a bottleneck (find best industry practices for these cases that are not so complex for an MVP)
        - Data privacy. So for this MVP the granularity we will offer is that the workflow creator is supposed to have permissions to access data from the surveys and pass it through a workflow pipeline. The permissions are per workspace, so a workflow is tied to the permissions of the workspace it is created. 1 Workspace -> N Workflows.

        We will need to decide the architecture behind: idempotency, how many workers we need, concurrency/parallelism, vertical/horizontal scalability, etc.

        ## Glossary

        (put this somewhere in workbench like e.g. business rules):

        **Trigger**
        The event that starts a workflow: record updated, survey response completed, schedule, manual run, webhook.

        **Action**
        A step that does something: send email, call webhook/API, update record, create task, delay. They have an input and an output, and the output can be structured.

        **Condition / Filter**
        A gate that decides whether execution continues. Example: only if respondent reached a specific thank-you card.

        **Branch**
        Multiple paths from one step. Useful for if/else behavior or parallel actions.

        **Workflow Definition**
        The saved source of truth: usually JSON/graph/schema with nodes, edges, configs, variables, and metadata.

        **Workflow Version**
        An immutable activated snapshot. Drafts can change, but active runs should execute against the version that was live.

        **Run / Execution**
        One concrete workflow execution after a trigger fires. Has status, inputs, outputs, timestamps, and errors.

        **Step Input / Output**
        Data passed into and returned from each step. Later steps reference earlier outputs.

        **Variables / Expressions**
        How users map data into fields, e.g. `recipient = response.contact.email`.

        **Idempotency**
        Protection against duplicate side effects when retries happen, e.g. not sending the same email twice.

        **Retry Policy**
        Rules for what happens when a step fails: retry, stop, skip, alert, or manual rerun.

        **Queue / Worker**
        Async execution layer for workflow runs, delays, emails, webhooks, and retries.

        **Run Log / Audit Log**
        Trace of what happened, useful for debugging, compliance, support, and trust.

        **Dry Run / Test Run**
        Safe execution with sample or selected data before activation.

        **Scope / Permissions**
        Who owns, edits, runs, and manages workflows. For Formbricks, workspace/environment/org scoping is a major decision.
